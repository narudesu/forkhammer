# CLI

Forkhammer includes a host-side CLI that talks to Supabase directly.

It runs outside Docker, so you can enqueue and inspect work from your machine while the worker stays inside the container.

## How it works

```text
forkhammer queue add / read / list
    |
    v
insert or select rows in Supabase
    |
    v
Dockerized worker reacts to new rows
    |
    v
worker writes result events back to Supabase
    |
    v
CLI reads the same event stream by issue key
```

The queue commands use the event table as the source of truth:

- `queue add` writes a `validate_issue_requested` event.
- `queue list` shows recent events and the issue keys they belong to.
- `queue read` reads the event history for one issue key and prints the latest result.

## Build

Build the CLI binary with:

```sh
bun run build:cli
```

Or build everything with:

```sh
bun run build
```

## Run

During development, run the CLI directly:

```sh
bun run ./src/cli.ts queue list
```

After building, run the compiled binary:

```sh
./dist/forkhammer queue list
```

## Common commands

```sh
forkhammer queue add AT-123
forkhammer queue list
forkhammer queue read AT-123
```

The CLI uses the same `~/.config/forkhammer/config.toml` file as the worker, so the Supabase URL, auth, and project mapping stay consistent.

## Peer session commands

The `peer` commands use the same resolver target as the PeerJS API. In addition to listing state, they can create worktrees and sessions, send one-shot prompts, stream summarized session events, and delete session files:

```sh
forkhammer peer create-worktree <project> <name>
forkhammer peer create-session <worktree-path> --name "Refactor auth module"
forkhammer peer prompt-session <session-path> "Inspect the current changes"
forkhammer peer prompt-session <session-path> "Implement the fix" --mode write
forkhammer peer subscribe-session <session-path>
forkhammer peer archive-session <session-path>
```

Subscriptions print user/assistant messages and tool-call summaries until interrupted. Project names are the TOML project record keys; an existing worktree is reported as an error rather than reused.
