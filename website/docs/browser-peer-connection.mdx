# JSON-RPC peer integration

Forkhammer uses **PeerJS** for signaling and a direct WebRTC data channel for communication. It uses [`json-rpc-2.0`](https://www.npmjs.com/package/json-rpc-2.0) on that channel to call worker methods.

## Connection flow

1. The browser creates a PeerJS peer and waits for its ID.
2. The browser writes a `browser_peer_ready` event to Supabase:

   ```json
   { "event_type": "browser_peer_ready", "data": { "peerId": "<browser-peer-id>" } }
   ```

3. The worker sees the event and connects to that peer ID.
4. On `connection`, the browser keeps the data connection open and sends JSON-RPC messages.

PeerJS is only used for signaling. After the connection opens, requests and responses travel directly between the two peers.

## JSON-RPC messages

The worker exposes these methods:

- `get-config`
- `list-worktrees`
- `list-sessions`
- `get-session`
- `create-worktree`
- `create-session`
- `subscribe-session`
- `unsubscribe-session`
- `archive-session`
- `prompt-session`

A request has an `id`, and its response uses the same ID:

```json
{ "jsonrpc": "2.0", "id": 1, "method": "list-worktrees", "params": { "project": "demo" } }
```

```json
{ "jsonrpc": "2.0", "id": 1, "result": { "project": "demo", "worktrees": [] } }
```

Failures are returned as a JSON-RPC `error` response. Method parameters and result shapes are defined in [`src/peer-protocol/peer-protocol.ts`](https://github.com/narudesu/forkhammer/blob/main/src/peer-protocol/peer-protocol.ts).

## Integrate the browser peer

Install the two client dependencies:

```bash
bun add peerjs json-rpc-2.0
```

A minimal browser-side client looks like this:

```ts
import Peer from "peerjs";
import { JSONRPCClient } from "json-rpc-2.0";

const peer = new Peer();

peer.on("open", async (peerId) => {
  // Insert this event using the application's Supabase client.
  await supabase.from("ultrafeed_item").insert({
    event_type: "browser_peer_ready",
    data: { peerId },
  });
});

peer.on("connection", (connection) => {
  connection.on("open", () => {
    const rpc = new JSONRPCClient((message) => connection.send(message));

    connection.on("data", (message) => {
      // The worker sends session updates as JSON-RPC notifications.
      if (message && typeof message === "object" &&
          (message as { method?: string }).method === "session-event") {
        handleSessionEvent((message as { params: unknown }).params);
        return;
      }
      void rpc.receive(message as never);
    });

    rpc.request("list-worktrees", { project: "demo" })
      .then((result) => console.log(result))
      .catch((error) => console.error(error));
  });

  connection.on("close", () => {
    // Mark the client disconnected and create/announce a new peer when needed.
  });
});
```

Use the same `JSONRPCClient` instance for all requests so it can match responses to request IDs. Keep the PeerJS peer alive, handle `connection`/`close`, and announce a new `browser_peer_ready` event after reconnecting.

## Session updates

After calling `subscribe-session`, the worker sends notifications (they have a `method` but no request ID):

```json
{
  "jsonrpc": "2.0",
  "method": "session-event",
  "params": { "sessionPath": "...", "type": "assistant_message", "text": "..." }
}
```

Handle these notifications separately; do not pass them to the request-response flow.
