# Where UI

In the initial hobby-project rapid development phase of forkhammer, I've decided to not tightly couple the UI and the worker itself.

**Why**


This allows me to open-source the core worker code and allow people to use it independently of UI I aim to change often to optimize to my needs.

If you'd like to use the UI while I'm developing it, it is available at https://grsq.netlify.app, along with other tools I'm developing as hobby projects to help me live life and work work.

The GRSQ app currently uses the Supabase free tier and it is also free.

**Disclaimer**: Availability of GRSQ might change based on Supabase free tier exhaustion and rapid development.

**Future Plans**

My primary goal is to polish the UI into a state where I'm happy with it and then open-source it to allow its usage in addition to forkhammer.

I'm currently unsure what the UI should look like and what features I want in it so for now it will be decoupled from forkhammer.

My brain dump:
- how much should be delegated to the existing OpenCode UI and how much of the features in OpenCode should this UI duplicate
- do we want to keep using OpenCode or is Pi Coding Agent more flexible for this usecase?
- do I want some automations such as automated ticket status change
- maybe I could implement some blocker management and git log watching with automated forkhammer input on unblocking
- maybe we could make it easy to split larger tasks into Jira tickets that can be later fed into forkhammer
- maybe it should provide centralized information such as: healthchecks, gitlab pipeline status, automated merge request creation and similar
- do I want to create UI for everything or is a Hermes agent with API access and skills a better fit?
